# 订单ID提取逻辑改进示例

## 改进前的问题

**原始逻辑问题：**
- 当无法提取到6位以上数字时，会使用文件名的ASCII部分作为订单ID
- 例如：`T3-77.06-1-行程单.pdf` → 订单ID: `T3`
- 这样的订单ID缺乏完整信息，用户无法了解订单的详细信息

## 改进后的效果

### 1. 阳光出行订单
**文件名：** `阳光出行-32.13元-3个行程-行程单.pdf`
**改进前：** 订单ID: `阳光出行` (不完整)
**改进后：** 订单ID: `阳光出行-32.13元-3个行程` ✅

**文件名：** `阳光出行-29.59元-1个行程-发票.pdf`
**改进前：** 订单ID: `阳光出行` (不完整)
**改进后：** 订单ID: `阳光出行-29.59元-1个行程` ✅

### 2. T3出行订单
**文件名：** `T3-77.06-1-行程单.pdf`
**改进前：** 订单ID: `T3` (不完整)
**改进后：** 订单ID: `T3-77.06-1` ✅

**文件名：** `T3-77.06-1-发票.pdf`
**改进前：** 订单ID: `T3` (不完整)
**改进后：** 订单ID: `T3-77.06-1` ✅

### 3. 高德打车订单
**文件名：** `高德打车-94.42元-2个行程-行程单.pdf`
**改进前：** 订单ID: `高德打车` (不完整)
**改进后：** 订单ID: `高德打车-94.42元-2个行程` ✅

## 改进的核心特性

### 1. 保持原始服务商名称
- ❌ 不再使用缩写：`YG`、`GD`、`DD`等
- ✅ 直接使用：`阳光出行`、`高德打车`、`T3`等

### 2. 包含完整订单信息
- **服务商名称**：用户一眼就能认出
- **订单金额**：便于财务核对（保留"元"字更自然）
- **行程数量**：了解订单规模（保留"个行程"更直观）

### 3. 支持多种文件命名格式
```
阳光出行-32.13元-3个行程  → 阳光出行-32.13元-3个行程
T3-77.06-1                → T3-77.06-1
高德打车-45.20元-2个行程   → 高德打车-45.20元-2个行程
滴滴出行-28.50元          → 滴滴出行-28.50元
```

**优势：**
1. **可读性更强**：用户可以直接看到完整的订单信息
2. **便于识别**：服务商、金额、数量一目了然
3. **减少混淆**：避免多个T3订单难以区分的问题
4. **财务友好**：金额信息直接显示在订单ID中
5. **表达自然**：保留"元"和"个行程"等自然表达方式

## 技术实现细节

### 正则表达式模式
```python
# 支持多种格式的正则表达式
r'([^-]+)-(\d+\.\d+)元?-(\d+)个行程'  # 阳光出行-32.13元-3个行程
r'([^-]+)-(\d+\.\d+)-(\d+)'           # T3-77.06-1
r'([^-]+)-(\d+\.\d+)'                 # 服务商-金额
```

### 智能提取逻辑
1. **优先匹配**：完整的"服务商-金额-数量"格式
2. **次优匹配**：简单的"服务商-金额"格式
3. **回退机制**：如果都失败，使用原有的ASCII提取方法

### 错误处理
- 如果PDF内容提取失败，不会影响文件名提取
- 如果XML解析失败，会记录警告但继续处理
- 最终回退到安全的文件名提取方法

## 总结

这次改进解决了以下问题：

1. **订单ID不完整**：从`T3`提升到`阳光出行-32.13元-3个行程`
2. **可读性差**：从缩写改为完整服务商名称
3. **信息缺失**：订单ID现在包含金额和数量信息
4. **用户困惑**：页面显示更加清晰明了
5. **表达不自然**：现在保持"元"和"个行程"等自然表达方式

改进后的订单ID既保持了技术上的唯一性，又大大提升了用户体验和可读性。特别是保持了中文表达的自然性，让用户感觉更加亲切和易懂。
